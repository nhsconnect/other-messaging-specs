Short Title of Issue

Batching schema changes as a result of issues raised by BT

Description

 

MIMC1592 : BT has asserted the position that the batch message should only contain MT type messages, not the full Interactions.  This has been asserted because of the overhead created by sending irrelevant information (from the wrapper) in every request/response.  On an average message this overhead is in the range of 1.2KB--1.4KB.  To make a rough calculation, if the average MT message size is 15KB, the current mechanism for batching would allow 300 messages in batch and contain a wrapper overhead of about 420KB.  Without this overhead it would be possible to fit more than 25 more records in a batch.  While these numbers may not appear significant, a design that allows this overhead for no practical reason should be re-factored. It is our understanding that the MT-style of design was taken to HL7 for comment, and they rejected the approach (stating that performance and efficiency was not an appropriate reason for modifying the standard).  It is our view that the National Program is driving HL7 standardisation in this area, that HL7 do not have a defined standard for batching, and that performance and efficiency are key non-functional design criteria of the Spine. As such we request that the National Program re-consider this HL7 design.  We are unable to accept a design that may affect our ability to meet our SLAs.

 

MIMC1606 : Model (MCCI_RM200001UK01) states that the cardinality of a message is 1..*.  However, the schema has the message (MCCI_MT200001UK01) defined as 0..*.  The schema needs to be corrected.
