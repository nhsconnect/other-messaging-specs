Short Title of Issue

Changes required to Batching for MIM 4.1.02 as a result of issues raised by BT

Description

 

MIMC1595&MIMC1594: On a Send Batch, Batch Seq No. needs only be unique within a concatenation of ebXML ConversationId and  Accedited System Id (DeviceId in the wrapper) and not mandatory on any batch.

 

MIMC1593: The MHS, the reciever responsibility reason is "batch recieved", this should be "Batch Response"

 

MIMC1596: It should be pointed out that the implementation of Batch Sequence Number is not the only mechanism required of a batch processor to determine missing batches.  This mechanism does not aid in the identification by the Receiver of a lost last Send Batch message.  The Receiver of a Send Batch has no method of determining where there is a next batch request in the conversation.  Hence it is the responsibility of the Sender to determine that batch requests have gone missing and have not been processed by the Receiver.

 

MIMC1600: Manifest Items:  This class is mandatory on any batch response from Spine

 

MIMC1599: On a  Send Batch the use of Manifest Items is probably not precise enough.  Suggest that Manifest Items must only be provided when domain specific guidance is given by the Authority

 

MIMC1597: The name ManifestItems should not be plural.  It represents a single manifest item.  This is a collection of 0..* ManifestItem.

 

MIMC1598: On A Send Batch Response, the ManifestItem does not need to carry the ConversationId.  This will be the same as the ConversationId in the ebXML Header, and hence is not necessary

 

MIMC1591 :Most of the examples for batching contain invalid manifest items; the message id should be of root and extension type.
