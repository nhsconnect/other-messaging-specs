1    General Information

1.1 Overview

 

PSIS Queries provide a mechanism for clinicians to gain access to the NHS Care Record Spine for an individual service user (patient). Local systems will query the service user’s NHS Care Record and populate screens with NHS Care Record data.

 

This version of PSIS-Query relates to PSIS 2008.

 

In this version the following queries and associated responses are specified:
 

  • PSIS Document List Query - requests a list of metadata about documents with optional parameters.
     
  • PSIS Document Query - data is requested for a single specified Document where the identifier has been retrieved using another query message. This is the only query which returns a message payload. Only one document can be queried on at a time. To query multiple documents, batching must be used.
     
  • PSIS CRE List Query - a list of documents containing a given CRE type is returned. When a parent CRE type is queried on metadata about documents of that CRE type and child CRE types will be returned. No indication of the CRE type will be returned in the list response. Metadata about GP Summaries will not be included in the response.
     
  • PSIS Synchronisation List Query - metadata is retrieved about all documents that been persisted on PSIS within the stated date/date-range parameter. All documents (active and inactive) will be returned with this query.

Further information on business flow for PSIS Query can be found in the Integration Strategy.
 

1.2 Batching

In order to process several document requests and responses, the batching solution as used by PDS should be implemented. This is detailed in the Infrastructure section of this MIM.

 

Batching is intended to be used with the Document Query and Response.

 

1.3 Back Office Admin Interactions

This version of PSIS Query is the first to include interactions which are not intended for Accredited System use. The PSIS-Admin interactions have been included in the MIM so that back office developers (associated with Spine applications) can reference the interaction, message specifications and IDs. New trigger events have also been created for these interactions.

 

The PSIS-Admin interactions will use existing queries however operate on a different group of documents on PSIS. Currently the Document and Document List queries have PSIS-Admin interactions. When used with the standard PSIS interactions these queries (except Synchronisation List query) work against only active documents, when used with PSIS-Admin interactions these queries work with both active and inactive documents.

 

Spine will reject the use of PSIS-Admin interaction from unauthorised systems. Authorised systems will consist of back-office administrative applications internal to the spine.

 

1.4 Schematron

A schematron is provided at \work\schematrons\PSIS_Query_Schematron.xml which validates the constraints illustrated in the PSIS Query 'request' messages with regards to dissent override values requiring the existence of certain authors. The inclusion of this schematron is for optional use and helps enforce the cross-wrapper constraints between the query control act wrapper and payload.

1.5    Known Issues

1.5.1 Context Conduction

PSIS will not support the HL7 concept of Context Conduction when indexing meta-data. It should be noted that PSIS will only index acts which are explicitly stated. Where they are inferred, e.g. for some authors in CDA documents at Section level, these will not be indexed or returned in queries for that section.

 

1.6 Authors and Control Acts

In order to remain consistent with other domains that contain queries, all queries now use the standard control acts from the Infrastructure domain. The PSIS Query RMIMs no longer illustrate the control acts as part of the query payload however they are included as part of the interaction - see Interaction tables in Section 6. A PowerPoint presentation is available which helps explain these concepts in \work\External Documents\PSIS Query\AuthorMappingPSIS.ppt.

2.1    Sally’s Follow-up Appointment with her GP

Sally attends her local GP surgery for an appointment; following on from an ED visit.

The GP selects on his local system to retrieve a list of events over the last year relating to Sally – the local system sends the request [PSIS Document List Query] to PSIS. The system presents a list of active documents received by PSIS in the past year [PSIS Document List Query Response]. The list contains a recent ED Report which shows the date and time the report was received by PSIS, the Document Type (i.e. Emergency Department Report), the name of the Author and the name of Organisation the letter originated from.

2.3    Patient Has No Records On PSIS

Dr Sugden has a new patient, Jane Robinson, who’s previous GP was not PSIS compliant. He gains the necessary LR to be able to access Jane's record.

Dr Sugden again uses his local system to request a list of Care Events from PSIS [PSIS Document List Query]. After confirming that the appropriate access rights exist, PSIS attempts to retrieve a list of all events recorded.

As Jane's previous GP was not PSIS compliant PSIS cannot find any documents for her and returns the appropriate error code to the local system [Query Act Failed] – with the appropriate error code(s).   

2.4    LSP System Synchronises with PSIS

A local system routinely identifies the need to refresh its local cache of PSIS National Summary Care Records. It identifies the date of the last synchronisation (for each of the patient’s record individually) between the spine and the local system.

The local system requests a list of documents with a received date or changed date between the last synchronisation date and the current local system date (for each patient in turn)  [PSIS Synchronisation List Query].

 

PSIS returns a list of meta-data about the requested documents to the requesting local system  [PSIS Synchronisation List Query Response].

 

The local system then uses the metadata to calculate the size of messages and send batches of document requests to return responses which are within the TMS size limit (~5MB).

8    Glossary of Terms

Active Document The state of 'Active' is a virtual state to classify documents on PSIS where the attributes/values are as listed below:
  • PSISDataStatus = "Normal'
  • DocumentDQState = "NORMAL" or "DQSUSPECT".
CDA The HL7 Clinical Document Architecture (CDA) is a document mark-up standard that specifies the structure and semantics of "clinical documents" for the purpose of exchange.
Clinical Document A clinical document is a documentation of clinical observations and services.
Dissent Dissent refers to an articulated decision by a patient, or their authorised representative, that they do not consent to the patient's sensitive personal data being shared, for the purpose of providing healthcare to that patient, outside the boundary of the legal organisation responsible for creating or maintaining them.
Document When referring to a document, this corresponds to the old concept of an Event. In PSIS a Document is any single message payload stored on PSIS. Examples include a GP-Summary (CSMP-based), or an Outpatient report (CDA-based).
DQ State Data Quality State - a Document on PSIS will normally have a DQ state of 'NORMAL' - this can be changed to a different state by Clinical Back Office applications only.
Event Historically referred to as a PSIS Event, the terminology has now been change to 'Document'. An Event (Document) is any single message payload stored on PSIS.
Event Item Historically, Event Items corresponded to Clinical Statements (either single or compound). Event items refer to information which is returned in PSIS Query Reponses at Document Section Meta Data level.
Message PSIS-Queries and responses are known as messages which return metadata about Documents and in some cases a complete Document. A message includes the wrappers.
Message Payload A Message Payload is carried in a message and excludes wrappers. A payload could be a Document or a query meta-data response.
PSIS Personal Spine Information System.
Query Originator Clinician or Caldicott Guardian initiating the query.
Section A Section is part of a CDA document which can contain any block below the CDA Header. CDA Sections can contain either:
i) A Business Section
ii) A CRE-Type Section
iii) A Text Section
iv) A coded Clinical Statement Section.
Service User The patient who is the subject of a query.

 

 

9    Conformance to HL7

All query request messages in the PSIS-Query domain conform to the HL7 Standard. The query response messages have been defined in a HL7-RIM conformant manner.
