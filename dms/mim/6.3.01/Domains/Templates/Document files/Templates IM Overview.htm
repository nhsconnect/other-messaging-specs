1    Overview

The templates defined in this manual are intended to ensure  consistent and tightly constrained communication of clinical information within messages.

 

1.1    Use of Templates in CDA domains

Message instances must validate against the CDA balloted schema, the NPfIT CDA constrained schema and the domain message schema and also any template schemas used. Message instances will use xml tag names (classnames)  as per the balloted CDA schema. Where the templateId holds a reference to a template then the message instance must validate against  the appropriate template schema. The appropriate transform should be run against the instance to transform the generalised instance tagnames to the template ( business ) names.

Interactions for CDA domains will be bound to the NPfIT CDA constrained schema.  

 

The following rules shall be followed when using templates:

  • A template for an act can be entered by an act relationship at the specified entry point, the template id will be held in the focal act of the template and all other classes within that template.

  • Alternatively a template for an act can be entered by an act relationship at the specified entry point, the contentId id will be held in that act relationship

  • A template for a role can only be entered by a participation at the specified entry point, the template id will be held in the focal act of the template and all other classes within that template.

  • Alternatively a template for a role can only be entered by a participation at the specified entry point, the contentId id will be held in that participation

  • All mandatory and required classes and attributes in the template shall be included and populated accordingly

  • Optional classes and attributes in the template may be included so long as they conform to the specification of that class/attribute

  • Outbound act relationships or participations contained in the template allow templates to be included within templates

  • Class clone names in the template models, tabular views are for illustration only and in message instances they shall comply with the names in the balloted CDA schema.

 

Further details of the NPfIT CDA implementation using templates refer to the latest version of  "NPFIT-ELIBR-AREL-P1R2-0178 Technical Guidance for Implementation of Templated CDA Domains" 

 

1.3    Use of Template in Non CDA domains

Message instances must validate against the domain message schema and also any template schemas used. Message instances will use xml tag names (classnames)  as per the domain and template schemas. Where the templateId holds a reference to a template then the message instance must validate against  the appropriate template schema.

Interactions for Non CDA domains that use templates will be bound to the domain schema in the normal way.  

 

The following rules shall be followed when using templates:

  • A template for an act can be entered by an act relationship at the specified entry point, the template id will be held in the focal act of the template and all other classes within that template.

  • Alternatively a template for an act can be entered by an act relationship at the specified entry point, the contentId id will be held in that act relationship

  • A template for a role can only be entered by a participation at the specified entry point, the template id will be held in the focal act of the template and all other classes within that template.

  • Alternatively a template for a role can only be entered by a participation at the specified entry point, the contentId id will be held in that participation

  • All mandatory and required classes and attributes in the template shall be included and populated accordingly

  • Optional classes and attributes in the template may be included so long as they conform to the specification of that class/attribute

  • Outbound act relationships or participations contained in the template allow templates to be included within templates
