Short Title of Issue

General use of IVL<TS> in place of GTS

 

Description

Formalise the tabular view comments that GTS is not to be used for P1R2. Generally replace GTS with IVL<TS> in PoC and medication messages.

Perceived Impact

 

 

Backward Compatibility Status

 

Reason for issue being raised

Suppliers (BT NASP, Accenture, BT LSP) have consistently requested that for P1R2 (and presumably early P2) medication and PoC messaging, the HL7 GTS datatype not be used, and that IVL<TS> be used in its place. Whilst this is alluded in tabular views, it is not formally accepted in the MiM and associated documentation. It is understood that IVL<TS> does not allow the representation of as rich a set of cases as GTS. But the HL7 ballot pack documentation on GTS is antihelpful, and NPfIT has not to date generated a set of examples of the use of GTS that would permit suppliers safely to understand how to use it.

Suggested Action

Identify all message definition instances (i.e. there is no need to touch CSMP for this) of GTS in ETP and PoC messages. Replace with IVL<TS> in the RMIMs. Refactor examples and tabular views accordingly.

 

CTT Resolution and Current Status

Proposed Change:

 

The following was decided at a PSIS ACRS Requirements workshop of 15-16 
February 2005.  This is an extract from the minutes.
{
•       The PSIS will only process a GTS where it is specialised to a SET<TS> 
or LIST<IVL<TS>>. This specialisation must be uniquely identifiable in the 
instance of the message. On identification of a GTS, the PSIS will undertake 
the following processing to generate a convex hull of time for use in an 
object effective time (for example);
•       The effective time of an object can be derived by taking the Set of 
Intervals of time stamps, that is the implementation for all GTS, and 
extracting the effective time low value as the earliest of all of the low or 
centre values, and the effective time high value as being the latest of all 
high or centre values.  Values with just a width will not contribute to the 
result.
•       All other types of GTS will be treated as absent for the purposes of 
query filtering.
•       A GTS may not be used as a supplied parameter in a query.
•       A GTS specialised as a SET<TS> or LIST<IVL<TS>> will not contain 
literals or their symbolic representation
}
 
THIS MEANS:
Wherever possible (and I would suggest that means everywhere for P1R2), GTS 
should not be specifed in messages but rather SET<TS> or LIST<IVL<TS>> should 
be used and it should be stated that literals, or their symbolic 
representation, must not be used.
 
The expectation is that the only exception to this in the future will arise 
when the medication dose syntax is fully introduced.  However it is expected 
that it will still be possible to derive the 'convex hull' (earliest start to 
latest end - to you and me), by the logic shown above.
 
At P1R2, medication messages should carry a SBADM EffectiveTime as a single 
IVL<TS> carrying the 'convex hull' as it would be derived from the full dose 
syntax.
 
Observation Acts can record multiple dates or multiple date ranges but each 
IVL<TS> in the LIST should use one of the flavours explicitly documented in 
the MiM.  I.e. they should not contain literals or symbolic elements.
