Short Title of Issue

Misc. Tabular View and Vocab Changes

Description

 

Bugzilla issue

 

10162

Parent Prescription uses the DispensingSitePreference vocabulary for nominated pharmacy information, and this vocabulary entry states that the values held here “reflects the PDS Code scheme for P1R2”.  Is it the intention that these values should be the same in both domains? As it stands at the moment these code schemes are different and conversion between the DispensingSitePreference PDS vocabulary (Patient Care Provider Type) will be necessary. 

13614

Correct usage of GTS in Parent Prescription - PORX_MT132004UK30 and Dispense Notification - PORX_MT024001UK30 unclear.

13938

The tabular view of all ETP messages with id LIST<II>[2..2] should include that the first id should be the UUID in order to create certainty and avoid confusion. PSIS will always refer to the first UUID when a LIST of Ids is present.

13932

Text for RequestedMaterial.code [1..1] code (CE {CNE:ManufacturedMaterialSnCT} ) in tabular view of following messages contradicts datatype specification:

7.1    Parent Prescription - PORX_MT132004UK30

7.11    Dispense Notification - PORX_MT024001UK30

7.16    PA Notification - PORX_MT114002UK30

7.21    PS Notification - PORX_MT024003UK30

13935

A fixed code (163541000000107) from the SnomedCT UK subset that describes Dispense Notification (PORX_MT024001UK30) as a Dispensed Medication event is used here. What is the value in using the "Coded with Original Text" datatype flavour? What more will it convey then "Medication Dispense"?

13931

FocalAct.code should be Coded Plain flavour not Coded with Original Text (what would original text be in this context?)

13947

CareRecordElementCategory.code should be Coded Plain flavour not Coded with Code System (this means any system displaying this data will have to find the displayName itself)

14111

Date format precision for Date of Death not mentioned. YYYYMMDD

14203

The DaysSupply class uses the expectedUseTime in what would appear to be

the wrong way.     From the decription of the use of this attribute it would

seem that it should use effectiveTime with the Date or Time Duration

Unanchored datatype flavour.

14204

The expectedUseTime attribute is present in the example as effectiveUseTime

14208

Why is the responsibleParty mandatory in Parent Prescription? The tabluar view states that this is for use when the author is not the responsible doctor.

14209

Why is the responsibleParty mandatory in Cancellation Request? The tabluar view states that this is for use when the author is not the responsible doctor.

14238

MIM-CR-0385 - MM Date formats All date formats will be YYYYMMDDHHMMSS

unless specified.  1. Deceased Time is not in Parent Prescription - yet it

is listed in CR. 2. ReviewDate.EffectiveTime uses YYYYMMDDHHMMSS but CR

says it should use YYYYMMDD.  Which is it?

14519

Two changes are required.

1) In the "DispensingSitePreference" vocab, please can we rename the

description for code "P3" from "Oxygen Contractor" to "Dispensing Doctor".

2) In the "PatientCareProvisionType" vocab, please can we rename the

description for code "P3" from "Oxygen supply" to "Dispensing Doctor".

14524

The tabular view description of the MessageRef element in the dispense, PS and

PA messages incorrectly states that the id references the "clinical id" of the

previous message.  The id should in fact reference the (focal act) id of the

previous message, as described in the CSMP.

14526

The BatchInfo element is incorrectly described in the

PrescriptionReleaseResponse tabular view. 

 

The message should contain 0 or 2 instances of the element, which in the

latter case represent:

1. the actual number of prescriptions in the batch and

2. the permitted maximum number of prescriptions in the batch. 

 

The descriptions of the codes in the NominatedBatchInfo vocab also are

incorrect and need amending as above.

 

Perceived Impact

Minimal impact – tabular view or vocab changes only required.

Reason for issue being raised

This CR has been raised to address a number of low impact issues raised by BT NASP and Accenture LSP review of MiM 4.0.01

Suggested Action
  1. Issue 10162 partly fixed in MiM 4.0, but mention of codes “0002 or 0003” in the description of “P1” needs to be removed to avoid any confusion.
  2. Issue 13614 – append following comment to affected attributes: “THIS ATTRIBUTE IS RESERVED FOR FUTURE USE AND SHOULD NOT BE USED IN THIS VERSION OF THE MESSAGE.”
  3. Issue 13938 - affected tabular views to be updated to state the first id must be the UUID.
  4. Issue 13932 – append following comment to affected attributes: “NOTE: this attribute uses the CE data type due to limitations in the current HL7 toolset, which prevent it from being modelled using the CD dataype.” (as for PoC A_SupplyMedication CMET)
  5. Issues 13931 and 13935 – change tabular view description of affected datatypes to Coded Plain datatype flavour in all messages.
  6. Issue 13947 – rejected
  7. Issues 14111 and 14238: This is an error in the original CR, however, the relevant tabular views should be updated to state format of DoD if used.
  8. Issues 14203 and 14204: fix example
  9. Issues 14208 and 14209: the responsible party is mandatory and always required. In some cases this will be a different person to the author and in some cases it will be the same person, however the business decided that the responsible party is always required even if they are the same as the author.
