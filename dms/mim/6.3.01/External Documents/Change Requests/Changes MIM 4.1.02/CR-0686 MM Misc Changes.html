MM 2.2 Misc. Tab View and Vocab Changes

 

issue Description:

Issue 13033

========

AdministrationSiteSnCT has a note indicating This area of coding is not well defined.  Further work is underway to develop a better defined hierarchy.

 

Supplier resolution: Develop a better defined hierarchy.

 

Issue 15018

========

Should inFulfillmentOf1 and inFulfillmentOf2 relationships be

mutually exclusive ?  In candidate PSIS ACRS functionality, the message will be rejected if both exist and reference statements belonging to different Medication objects.

 

Issue 16112

========

The tabular view for "author" within ParentPrescription currently say's the Person.id should be populated with the SDS identifier. To allow for PPA processing of prescriptions, this attribute must instead be populated by the prescriber's code issued by the PPA.

 

Issue 16113

========

Two additional reasons when a prescription would be not dispensed via the EPS need to be added into the MiM vocab.

1) "Illegal NHS prescription"

2) "Prescribed out of scope item"

 

Issue 16114

========

The example for "Quantity in Alternative Units" is incorrect.

 

Issue 16426

========

The RMIM gives the "withdrawal type" and "withdrawal reason" prescription annotations as "WT" and "WR", but the tabular view and the vocabulary page for prescription annotation says "PWT" and "PWR".

 

Issue 16427

========

In the MiM 4.1.01 ETP domain, the tabular view for the withdraw message is

contradictory, as the description of "pertinentInformation3" says:

 

 [1..1] pertinentWithdrawID (WithdrawID)

 The id of the root clinical event that is pertinent to this withdrawal.

 

whereas the documentation for pertinentWithdrawID says:

 

 [1..1] value (II)

 Root clinical event identifier that is universally unique across all records.

 

This shall be the 18 character Short Form Prescription Identifier where it exists.

 

The UUID prescription id for an ETP R1 prescription is the same as the clinical statement id, but the 18-character id for an R2 prescription is not the clinical statement id - the clinical statement id is carried in the first of the two "id" elements of the prescription.

 

Issue 16429

========

It has been noted in live that one supplier is generating parent prescriptions where the NHS number contains spaces.  This is having a knock-on effect that dispensing systems cannot process the incoming message as they are expecting a

numeric value.

 

I have verified with the demographics team that the NHS number should never contain spaces.

 

However, although the MIM specifies it should include numeric content, it does not define a datatype for this field which prohibits spaces

 

Following on from the MIM, the XML schemas (both the live and 'tightened' versions) do not check that the NHS number is numeric and contains no spaces.

 

Issue 16431

========

Incorrect description of recordTarget typeCode in CancellationResponse - description of typeCode reads "A code indicating that the patient's records will hold the information relating to this Dispense."

 

Issue 17843

========

The 'repeatNumber' attribute within the 'Prescription' entity is only required for repeat dispensing prescriptions. The current MiM states this is required also for repeat prescriptions.

 

The Spine processes repeat prescriptions in the same way as acute prescriptions therefore this information is not required for repeat prescriptions.
