    RESTful API Architecture

    The Interface Mechanism Phase 2 REST API is based on the HL7 FHIR (DSTU2) REST API. The following subsections detail differences from the HL7 FHIR specification.

    Support for Historical Versions of Resources

    Servers are not required to support the vread interaction. Servers that do support vread must return version-specific Content-Location and Location headers in responses.

    REST API Interactions

    Instance Level

    read

    Servers shall return an ETag header with the versionId and a Last-Modified header. Servers that support the vread operation shall return a Content-Location header in the response which is the full version specific url.

    vread

    Servers are not required to support the FHIR vread interaction. Servers that support this operation shall:

    • return an ETag header with the versionId and a Last-Modified header.
    • return a Content-Location header with the response which is the full version specific url.

    Compartment Based Access

    URI Title Included Resources Inclusion Criteria
    [Base URL/URI]/fhir/compartment/patient Patient Patient link

    Resource Patient

    FHIR Interactions

    Servers shall support the following FHIR RESTful API interactions:

    • read
    • search

    Search Parameters

    Servers shall support the following search parameters:

    • identifier
    • identifier=[system]|[code]

    The following table defines valid system and code values:

    System code
    urn:fhir.nhs.uk:id/NHSNumber An NHS number, e.g. 9434765919

    Example Requests

    The following is an example search URL (submitted using HTTP GET) where [Base URL/URI]/fhir is the service root URL:

    • [Base URL/URI]/fhir/Patient?identifier=urn:fhir.nhs.uk:id/NHSNumber|9434765919

    If a Patient resource for NHS number 9434765919 exists, the search returns a Bundle containing all Patient resource with the specified NHS number identifier.

    Patient id's and Patient resource id's

    Clients should not assume any implicit mapping between a Patient resource's logical id and the patient's NHS number or any other patient identifier. That is, clients should not infer a Patient resource's logical id from any of the patient's identifiers.

    Servers should not use NHS Number or any other patient identifier as the Patient resource logical id.
